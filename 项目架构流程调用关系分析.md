# 项目架构流程调用关系分析

## 一、项目概述

本项目是基于 **GD32F450VG (ARM Cortex-M4, 240MHz)** 的工业级 **IoT 智能网关 / DTU** 固件系统。采用 **FreeRTOS** 实时操作系统，通过 **lwIP** 协议栈实现以太网通信，集成 **GPRS/4G** 无线通信双网冗余，支持电力计量、摄像机管理、环境感知、远程 OTA 升级、日志上传等功能。

- **软件版本**: V1.0.1.20260501
- **硬件型号**: FN-ZJTY-PY0
- **设备类型**: 0x0400 (YX 市电版)

### 3.2 start_task (初始化任务) 流程

```
start_task()
  │
  ├─ save_init_function()                    // 存储模块初始化 (LittleFS)
  ├─ com_recevie_function_init()             // 通信接收缓冲区 + 帧队列初始化
  ├─ app_get_storage_param_function()        // 从 Flash 恢复全部配置参数
  ├─ log_init_function()                     // 日志模块初始化
  │
  ├─ enet_system_setup()                     // 以太网硬件初始化 (PHY)
  │   └─ if 成功:
  │       └─ lwip_comm_init()                //    lwIP 协议栈初始化 (最多重试5次)
  │           └─ 创建 lwip_link_thread        //    链路状态监控线程
  │   └─ if 失败:
  │       └─ 进入纯无线模式
  │
  ├─ send_task_init()                        // 发送模块初始化 (消息池+队列+发送任务)
  │
  ├─ ★ 创建各业务任务 (taskENTER_CRITICAL 保护) ★
  │   ├─ xTaskCreate(alarm_task, prio=17, stack=1024)
  │   ├─ xTaskCreate(det_task,   prio=12, stack=4096)
  │   ├─ xTaskCreate(app_task,   prio=16, stack=4096)
  │   ├─ xTaskCreate(eth_task,   prio=13, stack=1024)
  │   ├─ xTaskCreate(gsm_task,   prio=14, stack=1024)
  │   ├─ xTaskCreate(print_task, prio=2,  stack=512)
  │   └─ xTaskCreate(dtu_task,   prio=7,  stack=1024)
  │
  └─ vTaskDelete(StartTask_Handler)           // 自删除,释放初始化任务栈
```

## 四、FreeRTOS 任务体系

### 4.1 任务列表与优先级

| 任务名称 | 优先级 | 堆栈大小 | 功能描述 |
|---------|--------|---------|---------|
| **alarm_task** | 17 (最高) | 1024 | 故障检测与保护处理 |
| **app_task** | 16 | 4096 | 主应用任务 (通信收发/配置/存储/开关) |
| **gsm_task** | 14 | 1024 | 4G/GPRS 网络管理 |
| **eth_task** | 13 | 1024 | 以太网管理 (TCP/UDP/ONVIF/SNMP/Ping) |
| **det_task** | 12 | 4096 | 传感器数据采集 (电量/温湿度/姿态/按键) |
| **send_task** | 11 | 1024 | 消息发送 (优先级队列+重试) |
| **dtu_task** | 7 | 1024 | MPPT 太阳能控制器 Modbus 透传 |
| **print_task** | 2 (最低) | 512 | UDP 配置广播接收与响应 |
| **lwip_link** | 8 | 512 | lwIP 链路状态监控线程 |

## 五、各任务详细流程

### 5.1 alarm_task -- 故障检测任务 (优先级最高)

```
alarm_task_function()
  │
  └─ for(;;) 每 10ms 周期:
      ├─ alarm_elec_collection_param()     // 电气参数检测
      │   ├─ 电压保护检测 (过压/欠压)
      │   ├─ 电流保护检测 (过流)
      │   └─ 漏电保护检测 (MIU)
      │   └─ 异常时 → app_report_information_immediately()
      │
      ├─ alarm_net_collection_param()      // 网络参数检测
      │   ├─ 主网络状态检测
      │   ├─ 各摄像机网络状态检测
      │   └─ 异常时 → app_report_information_immediately()
      │
      ├─ alarm_sensor_collection_param()   // 传感器参数检测
      │   ├─ 温湿度超阈值检测
      │   ├─ 箱体姿态倾斜检测
      │   ├─ 防雷器/空开状态检测
      │   └─ 异常时 → app_report_information_immediately()
      │
      └─ FeedFwdgt() + vTaskDelay(10)
```

### 5.2 app_task -- 主应用任务

```
app_task_function()
  │
  └─ for(;;) 事件驱动 (帧队列阻塞等待, ≤10ms):
      ├─ app_task_save_function()           // 存储管理
      │   └─ 检测 save_flag.status 位掩码
      │       ├─ SAVE_FLAG_DEVICE_PARAM     → save_storage_device_parameter_function()
      │       ├─ SAVE_FLAG_LOCAL_NETWORK    → save_stroage_local_network()
      │       ├─ SAVE_FLAG_REMOTE_NETWORK   → save_stroage_remote_ip_function()
      │       ├─ SAVE_FLAG_UPDATE_ADDR      → save_stroage_update_addr()
      │       ├─ SAVE_FLAG_UPLOAD_ADDR      → save_stroage_http_upload_function()
      │       ├─ SAVE_FLAG_COM_PARAMETER    → save_stroage_com_param_function()
      │       ├─ SAVE_FLAG_CAREMA           → save_stroage_carema_parameter()
      │       ├─ SAVE_FLAG_THRESHOLD        → save_stroage_threshold_parameter()
      │       ├─ SAVE_FLAG_SNMP_OID         → save_stroage_snmp_oid_parameter()
      │       ├─ SAVE_FLAG_RESET            → 恢复出厂设置
      │       └─ SAVE_FLAG_RS485_CONFIG     → save_stroage_rs485_param_function()
      │
      ├─ com_deal_main_function()           // ★ 通信帧处理
      │   └─ 从帧队列取出 com_frame_t
      │       └─ 解析指令 (cmd 字段):
      │           ├─ CONFIGURE_xxx (配置指令) → 调用对应 com_deal_configure_xxx()
      │           │   ├─ 服务器IP/端口        → app_set_remote_network_function()
      │           │   ├─ 本地网络参数          → app_set_local_network_function()
      │           │   ├─ 摄像机配置            → app_set_camera_function()
      │           │   ├─ 阈值参数              → app_set_threshold_param_function()
      │           │   ├─ SNMP OID             → app_set_snmp_ip_function()
      │           │   ├─ 设备名称/ID           → app_set_device_param_function()
      │           │   ├─ 风扇/加热/补光参数     → app_set_fan_param_function() 等
      │           │   └─ 最终 → app_set_save_infor_function() 触发存储
      │           │
      │           ├─ CR_QUERY_xxx (查询指令) → app_set_com_send_flag_function()
      │           ├─ CR_xxx_RESET (复位指令) → eth_set_network_reset() / gsm_set_module_reset()
      │           ├─ CONTROL_xxx (控制指令)  → app_set_switch_control_function()
      │           ├─ CONFIGURE_UPDATE_SYSTEM → update_set_update_mode()
      │           ├─ CONFIGURE_UPLOAD_FILE   → upload_set_upload_mode()
      │           └─ CONFIGURE_NOW_TIME      → app_set_current_time()
      │
      ├─ app_com_send_function()            // ★ 通信发送主逻辑
      │   ├─ app_com_time_function()        //    定时器管理
      │   │   ├─ 心跳计时 → COM_FLAG_HEART_PACK
      │   │   └─ 上报计时 → COM_FLAG_REPORT_NORMALLY_MASK
      │   │
      │   ├─ app_deal_com_flag_function()   //    标志位处理与组包
      │   │   ├─ COM_FLAG_HEART_PACK        → com_heart_pack_function() → send_msg
      │   │   ├─ COM_FLAG_REPORT_NORMALLY   → com_report_normally_function() → send_msg
      │   │   ├─ COM_FLAG_REPORT_IMMEDIATELY→ com_report_normally() (高优先级) → send_msg
      │   │   ├─ COM_FLAG_QUERY_CONFIGURATION→ com_query_configuration() → send_msg
      │   │   ├─ COM_FLAG_VERSION           → com_version_information() → send_msg
      │   │   ├─ COM_FLAG_IPC_INFO          → 触发 ONVIF 搜索 → com_ipc_device_information()
      │   │   ├─ COM_FLAG_SNMP_PARAM        → 触发 SNMP 查询
      │   │   ├─ COM_FLAG_PING_INFO         → 触发远程 Ping
      │   │   └─ COM_FLAG_CONFIG_RETURN     → com_ack_function() (回复 ACK)
      │   │
      │   └─ app_server_link_status_function()  // 服务器连接状态检测
      │
      ├─ app_open_exec_task_function()      // 开关执行 (补光灯/加热器定时)

      ├─ app_sys_operate_relay()            // 系统操作继电器 (远程控制)
      │
      └─ 每秒执行:
          ├─ RTC_Get_Time()                 // 同步时间
          └─ 内存利用率统计
```

### 5.4 eth_task -- 以太网管理任务

```
eth_task_function()
  │
  └─ for(;;) 每 20ms 周期:
      │
      ├─ eth_reap_stopped_tasks()           // 回收已停止的子任务 TCB
      │   ├─ tcp_client_delete()
      │   ├─ udp_multicast_delete()
      │   ├─ onvif_udp_delete()
      │   ├─ onvif_tcp_delete()
      │   ├─ snmp_delete()
      │   ├─ update_lwip_delete()
      │   └─ upload_lwip_delete()
      │
      ├─ eth_network_reset_function()       // 网络复位处理
      │   └─ if g_lwipdev.reset:
      │       ├─ 复位 TCP / UDP组播 / ONVIF_UDP / ONVIF_TCP / SNMP
      │       ├─ lwip_stop_function()
      │       ├─ enet_hard_reinit()         //  PHY 硬件复位
      │       └─ if 网线在位:
      │           └─ lwip_start_function()  //  重新启动 lwIP
      │
      ├─ eth_connect_control_function(UDP_MULTICAST)  // UDP 组播控制
      │   └─ udp_multicast_start_function()
      │
      ├─ eth_snmp_connect_function()        // SNMP 按需创建
      │   └─ if snmp_start:
      │       └─ snmp_start_function()      //  启动 SNMP 查询任务
      │
      ├─ eth_carema_search_function()       // ★ ONVIF 摄像机搜索 ★
      │   └─ 状态机: ETH_ONVIF_INIT → ETH_ONVIF_START → ETH_ONVIF_OSD → ETH_ONVIF_END
      │       ├─ ETH_ONVIF_START: onvif_udp_start()         (UDP 广播搜索)
      │       └─ ETH_ONVIF_OSD:  onvif_tcp_client_init()     (TCP 连接查询 OSD)
      │
      ├─ lwip_ping_multi_poll()             // 标准 ICMP Ping 轮询
      │
      ├─ eth_ping_remote_ip_function()      // 远程指定 IP Ping
      │
      ├─ 摄像机搜索模式分支:
      │   ├─ 模式1 (ONVIF):  rtsp_thread_stop()
      │   ├─ 模式2 (端口扫描): port_scan_detection_function()
      │   └─ 模式3 (RTSP):    rtsp_thread_start()
      │
      ├─ eth_tcp_connect_control_function() // TCP 客户端管理
      │   └─ 状态机:
      │       ├─ NO_CONNECT → INIT_CONNECT (DNS解析 → tcp_client_start)
      │       ├─ INIT_CONNECT → CONNECT (连接成功) / 超时复位
      │       └─ CONNECT 看门狗
      │
      ├─ eth_ping_timer_function()          // 巡检 Ping 计时
      ├─ eth_ping_detection_function()      // 主机/备机/摄像机 Ping 巡检
```

## 六、com 数据接收处理 -- 完整流程

### 6.1 协议帧格式

每条完整帧的二进制结构如下（以含设备类型的分支为例）：

```
┌──────┬──────┬──────┬──────┬──────┬──────┬──────────┬──────┬──────────┬──────┬──────┐
│ 帧头  │ 帧头  │ 版本  │设备类型│设备类型│  ID  │   ID     │ 命令  │  QN      │ 长度  │ 数据  │
│ 0xF0 │ 0xF0 │   1B │   HB  │  LB  │ 3B  │  (续)    │  1B │    8B    │  1B │ N字节 │
├──────┴──────┼──────┴──────┴──────┼──────┴──────────┼──────┴──────────┼──────┼──────┤
│             │                  │                 │               │      │      │
└─────────────┴──────────────────┴─────────────────┴───────────────┴──────┴──────┘
                                                                              │
                                     ┌───────────────────────────────────────┘
                                     ▼
                              ┌──────┬──────┐
                              │ CRC8 │ 帧尾  │ 帧尾  │
                              │  1B  │ 0xFF │ 0xFF │
                              └──────┴──────┴──────┘
```

- 帧头: `0xF0F0` (接收端) / `0x0F0F` (发送端)
- 帧尾: `0xFFFF`
- 校验: CRC8 (从版本号到数据体末尾)
- 固定开销(不含设备类型): 19 字节 = 帧头(2) + 版本(1) + ID(3) + 命令(1) + QN(8) + 长度(1) + CRC(1) + 帧尾(2)
- 固定开销(含设备类型): 21 字节 = 上述 + 设备类型字段(2)

### 6.2 核心数据结构

```c
// com.h - 每通道独立的组帧状态机(静态分配, 一通道一份)
typedef struct {
    uint16_t pos;                 // 当前帧写入位置
    uint16_t head;                // 帧头滑动窗口(2字节)
    uint16_t tail;                // 帧尾滑动窗口(2字节)
    uint16_t body_size;           // 数据体长度字段(从帧中偏移提取)
    uint32_t last_tick;           // 上次收到字节的时刻(半截帧超时用)
    uint8_t  buf[COM_FRAME_MAX];  // 帧组装缓冲区(最大100字节)
} com_parser_t;

static com_parser_t sg_parser[COM_CH_MAX];   // TCP和GSM各一份
static QueueHandle_t sg_rx_q;                 // 组满帧队列(深度4)
```

### 6.3 初始化: com_recevie_function_init()

```
com_recevie_function_init()
  │
  ├─ memset(sg_parser, 0, sizeof(sg_parser))     // 清零两个通道的组帧状态机
  └─ sg_rx_q = xQueueCreate(4, sizeof(com_frame_t))  // 创建帧队列(深度4)
```

### 6.4 生产者: com_rx_feed() -- 逐字节滑动窗口组帧状态机

这是整个通信模块最核心的函数。TCP 和 GSM 通道各自将收到的原始字节喂入，每个通道有独立的 `sg_parser[ch]` 状态机，互不干扰。

```
com_rx_feed(ch, data, len)
  │
  ├─ 参数校验: ch >= COM_CH_MAX || data == NULL || len == 0 → return
  │
  ├─ 获取通道专属解析器: p = &sg_parser[ch]
  │
  ├─ ★ 半截帧超时保护 ★
  │   └─ if (pos != 0                            // 正在组帧中
  │       && HAL_GetTick()-last_tick > 1000ms):   // 超过1秒未收到新字节
  │       └─ pos = 0;  // 丢弃半截帧, 复位状态机
  │
  └─ for (i = 0; i < len; i++):   // ★ 逐字节状态机 ★
      │
      ├─ 更新帧头滑动窗口: head = (head << 8) | data[i]
      │
      ├─ if (pos == 0):  // 未开始组帧, 等待帧头匹配
      │   ├─ if (head != 0xF0F0): continue;     // 未匹配, 继续滑动
      │   └─ if (head == 0xF0F0):                // 帧头匹配!
      │       ├─ buf[pos++] = 0xF0;              //   写入帧头首字节
      │       └─ body_size = 0;                  //   清零长度
      │       └─ 本字节会在下一次循环中被 buf[pos++] = d 写入
      │
      ├─ ★ 在固定偏移位置提取数据体长度字段 ★
      │   ├─ 含设备类型: if (pos == 9+8) body_size = d;
      │   └─ 不含设备类型: if (pos == 7+8) body_size = d;
      │
      ├─ if (pos < COM_FRAME_MAX):
      │   └─ buf[pos++] = d;                    // 逐字节写入缓冲区
      │   else:
      │       └─ pos = 0; tail = 0; body_size = 0;  // 溢出则丢弃整帧
      │
      ├─ 更新帧尾滑动窗口: tail = (tail << 8) | d
      │
      ├─ ★ 帧尾匹配 + 长度一致性校验 → 组满一帧 ★
      │   └─ if (tail == 0xFFFF
      │       && body_size+固定开销 <= pos):
      │       ├─ frame.src = ch;               // 记录来源通道
      │       ├─ frame.len = pos;
      │       ├─ memcpy(frame.buf, buf, pos);
      │       ├─ xQueueSend(sg_rx_q, &frame, 0); // 非阻塞投递到帧队列
      │       └─ pos = 0; tail = 0; body_size = 0;  // 复位状态机
      │
      └─ ★ 协议出错: 已达预期长度却无帧尾 ★
          └─ if (body_size+固定开销 <= pos):     // 已超预期长度
              └─ pos = 0; tail = 0; body_size = 0;  // 丢弃此帧
```

### 6.7 关键设计要点

| 设计点 | 说明 |
|--------|------|
| **滑动窗口组帧** | 每字节更新 `head`/`tail` 两个 16-bit 滑动窗口，无需 `memmem` 查找，适合嵌入式 |
| **通道隔离** | 每个通道独立的 `com_parser_t`，互不干扰，天然支持 TCP/GSM 并发接收 |
| **半截帧超时** | 流中断超过 1 秒自动丢弃，防止状态机永久僵死 |
| **非阻塞投递** | `xQueueSend(..., 0)` 队列满时丢弃帧，防止生产者阻塞 |
| **事件驱动消费** | `xQueueReceive(..., 10ms)` 阻塞等待，帧一到即处理，零轮询 CPU 开销 |
| **原路返回** | `frame->src` 记录来源通道，ACK/查询响应从同一通道回复 |
| **广播豁免** | `CONFIGURE_NOW_TIME(0xB1)` 命令豁免设备 ID 校验，支持广播校时 |

## 八、日志存储模块 (log) -- 完整流程

### 8.1 核心设计理念

日志模块基于 **LittleFS 文件系统 + SPI Flash**，采用 **双批次乒乓架构** (current + pending)，实现写入与上传完全解耦：

- **current 批次**: 活跃写入批次，文件名为固定 `device.txt`，状态为 `WRITING`
- **pending 批次**: 待上传批次（最多同时存在 1 个），状态为 `PENDING`
- 写入时 current 达到 5000 条上限 → 自动轮转为 pending → 创建新 current
- 上传前 pending 文件名最终化为时间戳格式
- 上传成功 → 删除 pending 释放空间；上传失败 → 保留 pending 等待重试

### 8.2 目录与文件结构

```
SPI Flash (LittleFS)
  │
  ├─ /log/
  │   ├─ current/                      ← 活跃写入批次
  │   │   ├─ batch.meta                ← 批次元数据文件 (log_batch_meta_t)
  │   │   │   ├─ magic: 0x4C4F474D    ← 魔数标识
  │   │   │   ├─ version: 0x0001       ← 版本号
  │   │   │   ├─ status: WRITING(1)    ← 批次状态
  │   │   │   ├─ total_count            ← 当前总记录数
  │   │   │   ├─ next_index             ← 下一条记录索引号
  │   │   │   ├─ create_timestamp       ← 批次创建时间戳
  │   │   │   ├─ last_timestamp         ← 最后一条记录时间戳
  │   │   │   ├─ create_time[15]        ← 创建时间文本 (YYYYMMDDHHMMSS)
  │   │   │   └─ log_file[64]           ← 日志文件名
  │   │   │
  │   │   ├─ catalog.idx               ← 目录索引文件 (log_catalog_item_t[])
  │   │   │   每条记录: {magic, index, timestamp, record_offset, payload_len, checksum}
  │   │   │   作用: 按时间/索引快速定位日志记录在数据文件中的偏移
  │   │   │
  │   │   └─ device.txt                ← 日志数据文件 (log_record_header_t + payload)[]
  │   │       每条记录: {magic(0x4C4F4752), index, timestamp, payload_len, reserved, checksum} + payload
  │   │       payload 最大长度: LOG_QUERY_PAYLOAD_MAX (2048 字节)
  │   │
  │   └─ pending/                      ← 待上传批次 (最多1个)
  │       ├─ batch.meta                ← status: PENDING(2)
  │       ├─ catalog.idx
  │       └─ device.txt  (上传前重命名为: {设备ID}_log_{时间戳}.txt)
```

### 8.3 全局状态变量

```c

// 流式读取(用于 HTTP 分片上传):
typedef struct {

    uint32_t   payload_remain;                           // 当前 payload 剩余未读字节

```

### 8.4 初始化流程: log_init_function()

```
log_init_function()

  │
  └─ log_do_init()                    // ★ 核心初始化 ★

      ├─ 1) log_mutex_init()          //   创建互斥信号量

      │   └─ log_meta_load(pending/batch.meta, &s_pending_meta)

      │
```

### 8.5 写入流程: log_device_write()

```
log_device_write(data, len)           // ★ 外部调用入口 ★
  │
  └─ log_write_common(data, len)
      │
      ├─ log_mutex_lock()
      │
      ├─ ★ 懒初始化 ★
      │   └─ if !s_log_inited: log_do_init()
      │
      ├─ ★ 前置检查 ★
      │   └─ log_current_meta_ready():
      │       ├─ s_current_meta.magic == LOG_META_MAGIC
      │       ├─ s_current_meta.status == LOG_BATCH_STATUS_WRITING
      │       └─ s_current_meta.log_file[0] != '\0'
      │
      ├─ ★ 写入前预轮转检测 ★
      │   └─ if total_count >= LOG_MAX_COUNT (5000):
      │       ├─ if s_pending_valid != 0:
      │       │   └─ return LOG_ERR_STORAGE_FULL  // 前一批尚未上传, 拒绝写入
      │       └─ else:
      │           └─ log_switch_current_to_pending()  ★ 主动轮转 ★
      │
      ├─ was_pending = s_pending_valid  // 记录写入前 pending 状态
      │
      ├─ ★ 追加日志记录: log_append_record(file_path, payload, len) ★
      │   ├─ 1) log_get_time() → RTC → timestamp + time_text
      │   │
      │   ├─ 2) 构造 log_record_header_t:
      │   │   ├─ magic       = 0x4C4F4752
      │   │   ├─ index       = s_current_meta.next_index
      │   │   ├─ timestamp   = RTC 时间戳 (epoch)
      │   │   ├─ payload_len = len
      │   │   ├─ reserved    = 0
      │   │   └─ checksum    = calc(header_fields + payload)
      │   │       └─ 算法: checksum = (checksum * 131) + data[i] (累加)
      │   │
      │   ├─ 3) 追加写入数据文件: current/device.txt
      │   │   └─ lfs_file_open(APPEND) → lfs_file_write(header + payload) → lfs_file_close()
      │   │
      │   ├─ 4) 追加目录索引: current/catalog.idx
      │   │   └─ lfs_file_open(APPEND) → lfs_file_write(catalog_item) → lfs_file_close()
      │   │       └─ if 写入失败: lfs_file_truncate() 回滚数据文件
      │   │
      │   └─ 5) 更新内存元数据 + 持久化 batch.meta:
      │       ├─ s_current_meta.total_count++
      │       ├─ s_current_meta.next_index++
      │       ├─ s_current_meta.last_timestamp = now
      │       └─ log_meta_save(current/batch.meta, &s_current_meta)
      │
      ├─ ★ 写入后自动轮转检测 ★
      │   └─ log_auto_rotate_if_needed():
      │       └─ if total_count >= LOG_MAX_COUNT && !s_pending_valid:
      │           └─ log_switch_current_to_pending()  ★ 触发轮转 ★
      │
      ├─ ★ 轮转通知 ★
      │   └─ if (was_pending == 0 && s_pending_valid != 0):
      │       └─ (解锁后调用) s_rotate_notify_cb()  // 通知上层有新批次待上传
      │
      └─ log_mutex_unlock()
```

### 8.6 自动轮转机制: log_switch_current_to_pending()

这是日志模块最核心的函数，通过 **LittleFS 文件重命名** 实现批次原子迁移：

```
log_switch_current_to_pending()
  │
  ├─ 1) 前置校验:
  │   ├─ s_pending_valid != 0? → return LOG_ERR_STORAGE_FULL (已有待上传批次)
  │   ├─ s_current_meta.status != WRITING? → return LOG_ERR_NOT_READY
  │   └─ s_current_meta.total_count == 0? → return LOG_ERR_NO_DATA
  │
  ├─ 2) 准备 pending 目录:
  │   ├─ log_create_dir(/log/pending/)
  │   └─ if pending 目录非空: return  (防止覆盖未上传数据)
  │
  ├─ 3) ★ 状态切换 (先持久化 meta) ★
  │   ├─ s_current_meta.status = LOG_BATCH_STATUS_PENDING
  │   └─ log_meta_save(current/batch.meta)
  │       └─ if 写入失败: 回滚 status=WRITING → 返回错误
  │
  ├─ 4) ★ 三步文件迁移 (lfs_rename) ★
  │   │
  │   ├─ Step 1: rename current/device.txt → pending/device.txt
  │   │
  │   ├─ Step 2: rename current/catalog.idx → pending/catalog.idx
  │   │   └─ 失败: 回滚 Step1 (pending→current) + meta 状态回滚
  │   │
  │   └─ Step 3: rename current/batch.meta → pending/batch.meta
  │       └─ 失败: 回滚 Step2 + Step1 + meta 状态回滚
  │       └─ 失败: lfs_remove(pending/*) 清理残留
  │
  ├─ 5) ★ 内存状态迁移 ★
  │   ├─ memcpy(&s_pending_meta, &s_current_meta, ...)
  │   ├─ s_pending_valid = 1
  │   └─ memset(&s_current_meta, 0, ...)
  │
  └─ 6) ★ 创建新 current 批次 ★
      └─ log_create_current_batch()
          ├─ 构造 s_current_meta: {magic, version=1, status=WRITING, create_timestamp, ...}
          ├─ 生成唯一日志文件名 (含设备ID + 时间)
          ├─ log_meta_save(current/batch.meta)
          ├─ log_create_empty_file(current/device.txt)
          └─ log_create_empty_file(current/catalog.idx)
```

### 8.9 流式读取: 供 HTTP 分片上传使用

这是一套为 `upload_http` 模块设计的流式读取接口，将 pending 批次全部 payload 按顺序输出为明文字节流：

```
log_open_pending_text_stream()
  │
  ├─ log_mutex_lock()
  ├─ log_text_stream_reset()           // 关闭之前可能残留的文件句柄
  │
  ├─ if !s_pending_valid: return LOG_ERR_NO_PENDING
  │
  ├─ 构造路径:
  │   ├─ lfs_file_open(pending/catalog.idx, RDONLY) → s_pending_text_stream.catalog_fp
  │   └─ lfs_file_open(pending/{log_file}, RDONLY) → s_pending_text_stream.log_fp
  │
  ├─ s_pending_text_stream.opened = 1
  └─ log_mutex_unlock()
      │
      ▼
log_read_pending_text_stream(buf, buf_size, &read_len)
  │
  ├─ log_mutex_lock()
  │
  └─ while filled < buf_size && stream.opened:
      │
      ├─ if payload_remain == 0:       // ★ 加载下一条记录的 payload ★
      │   └─ log_text_stream_load_next_payload()
      │       ├─ lfs_file_read(catalog_fp, &item, sizeof(item))
      │       │   └─ 跳过无效项 (magic/checksum 不匹配)
      │       ├─ lfs_file_seek(log_fp, item.record_offset)
      │       ├─ lfs_file_read(log_fp, &header, sizeof(header))
      │       │   └─ 校验 magic/index/timestamp/payload_len/checksum
      │       ├─ lfs_file_read(log_fp, payload_buf, header.payload_len)
      │       └─ payload_remain = header.payload_len
      │           payload_off = 0
      │       └─ 返回 1(成功) / 0(读完) / <0(错误)
      │
      ├─ copy_len = min(payload_remain, buf_size - filled)
      ├─ memcpy(buf + filled, payload_buf + payload_off, copy_len)
      ├─ payload_off += copy_len
      ├─ payload_remain -= copy_len
      └─ filled += copy_len
      │
      └─ *read_len = filled; return LOG_OK
      │
      ▼
log_close_pending_text_stream()
  │
  ├─ log_mutex_lock()
  ├─ lfs_file_close(catalog_fp) + lfs_file_close(log_fp)
  ├─ memset(&s_pending_text_stream, 0, ...)
  └─ log_mutex_unlock()
```

### 8.11 关键数据校验链

```
日志记录完整性保护 (三重校验):
  │
  ├─ batch.meta
  │   └─ magic(0x4C4F474D) + version(0x0001) + status 校验
  │
  ├─ catalog.idx (每条目录项)
  │   └─ magic(0x4C4F4743) + checksum(累加算法)
  │       └─ checksum: 对 item 所有字段(除checksum外)计算 checksum(131*n + data[i])
  │
  └─ device.txt (每条记录)
      └─ magic(0x4C4F4752) + checksum(header所有字段 ^ payload累加)
          └─ 初始化恢复时逐条校验, 截断损坏的尾部

错误码体系:
  LOG_OK              (0)   : 操作成功
  LOG_ERR_MUTEX       (-1)  : 互斥锁创建/获取失败
  LOG_ERR_INIT        (-2)  : 模块初始化失败
  LOG_ERR_INVALID_PARAM(-3) : 非法参数
  LOG_ERR_NOT_READY   (-4)  : 批次不可写或不可操作
  LOG_ERR_STORAGE_FULL(-5)  : 已有 pending 批次, 存储空间满
  LOG_ERR_FS          (-6)  : LittleFS 文件系统操作失败
  LOG_ERR_APPEND      (-7)  : 追加写入失败
  LOG_ERR_ROTATE      (-8)  : current→pending 轮转失败
  LOG_ERR_NO_PENDING  (-9)  : 无待上传批次
  LOG_ERR_NO_DATA     (-10) : 无可封存或可上传数据
  LOG_ERR_CREATE_BATCH(-11) : 新批次创建失败
  LOG_ERR_RECOVER     (-12) : 异常批次恢复失败
  LOG_ERR_MEMORY      (-13) : 查询内存申请失败
  LOG_ERR_DATA_TOO_LARGE(-14): 记录长度超过查询缓冲
```

### 11.2 有线 OTA 完整流程 (update_lwip)

```
update_lwip_task()                    // FreeRTOS 后台任务 (prio=9, stack=4096)
  │
  ├─ update_set_update_addr()         // 从系统配置同步最新 IP/端口
  │
  ├─ ★ 步骤1: 获取 info.txt (GET /{HARD_NO_STR}/info.txt HTTP/1.1)
  │   └─ http_update_get_info_txt_by_lwip(&ip, port)
  │       ├─ netconn_new(NETCONN_TCP) → netconn_connect()  (最多重试3次)
  │       ├─ 发送 HTTP GET 请求
  │       ├─ http_update_recv_http_response()  循环接收
  │       │   ├─ 每包数据 → http_update_save_response()  (追加到2KB缓冲)
  │       │   └─ http_update_check_response_completed()  (Content-Length完整性检查)
  │       └─ 解析 JSON: 提取 "version" 和 "url"
  │           ├─ ret = 0:  需要升级
  │           ├─ ret = 2:  已是最新版本, 跳过
  │           └─ ret < 0:  获取失败
  │       └─ http_update_close_connect_by_lwip()
  │
  ├─ ★ 步骤2: 获取 crc.bin 文件大小 (HEAD 请求)
  │   └─ http_update_get_crc_bin_file_size_by_lwip()
  │       ├─ netconn_new → netconn_connect()
  │       ├─ 发送 HTTP HEAD 请求
  │       ├─ 接收响应头 → 解析 Content-Length
  │       ├─ 校验: file_size 必须是 UPDATE_CHUNK_SIZE 的整数倍
  │       ├─ 计算: section_len = UPDATE_CHUNK_SIZE - 2  (每片有效数据, 去掉2字节CRC)
  │       ├─ 计算: section_total = file_size / UPDATE_CHUNK_SIZE
  │       └─ http_update_close_connect_by_lwip()
  │
  ├─ ★ 步骤3: 分片下载 crc.bin 数据 (GET Range 请求, 循环)
  │   └─ http_update_get_crc_bin_file_data_by_lwip()
  │       │
  │       └─ for (section_current = 0; section_current < section_total;):
  │           │
  │           ├─ netconn_new → netconn_connect()
  │           │
  │           ├─ 构造 Range 请求:
  │           │   └─ GET /{url} HTTP/1.1
  │           │       Range: bytes={start}-{end}
  │           │       (start = section_current * UPDATE_CHUNK_SIZE)
  │           │       (end = start + UPDATE_CHUNK_SIZE - 1)
  │           │
  │           ├─ 接收 HTTP 206 Partial Content 响应
  │           │
  │           ├─ http_update_parse_crc_bin_data()  // ★ 核心: 校验+写入 ★
  │           │   ├─ 验证 HTTP 206 状态码
  │           │   ├─ 验证 Content-Length == UPDATE_CHUNK_SIZE
  │           │   ├─ 定位 HTTP Body (跳过 \r\n\r\n)
  │           │   ├─ ★ CRC16-MODBUS 校验 ★
  │           │   │   ├─ 对 Body 前 (UPDATE_CHUNK_SIZE-2) 字节计算 CRC16
  │           │   │   ├─ 与 Body 末尾 2 字节存储的校验值比对
  │           │   │   └─ 不匹配 → 返回 -5 (上层重试当前分片)
  │           │   │
  │           │   └─ ★ 写入 SPI Flash ★
  │           │       ├─ 目标地址: UPDATA_SPIFLASH_ADDR + section_current * section_len
  │           │       └─ sf_WriteBuffer(buf, addr, section_len)  (在临界区中)
  │           │
  │           ├─ section_current++  // 推进分片索引
  │           └─ http_update_close_connect_by_lwip()
  │
  ├─ ★ 完成: http_update_success_reboot()
  │   ├─ 构造 BOOT_UPDATE_PARAM: {is_update=1, section_size, section_count, ...}
  │   ├─ update_write_boot_param() → sf_WriteBuffer() 写入 SPI Flash 最后1KB
  │   ├─ lfs_unmount() 卸载文件系统
  │   └─ app_system_softreset() → NVIC_SystemReset()  软复位
  │
  └─ 失败处理:
      ├─ http_update_failed()     → 写入 UPDATE_FAILED 到 Boot 参数区
      ├─ app_set_reply_parameters_function()  → 通知平台升级失败
      └─ http_update_clear_param()  → 清除 Flash 状态
      └─ vTaskSuspend(NULL)       // 不自删除, 由 eth_task 调用 update_lwip_delete() 回收
```

```
      │
      ├─ ★ 注入传输层回调, 调用 upload_http 内核 ★
      │   └─ upload_http_file_function(&request, &s_upload_lwip_transport, NULL)
      │
      │   s_upload_lwip_transport:  (有线 TCP 传输层实现)
      │   ├─ connect_fn: upload_lwip_transport_connect()
      │   │   └─ netconn_new(NETCONN_TCP) → netconn_connect()  (最多3次重试)
      │   │       └─ tcp_nagle_disable()  // 禁用Nagle, 避免小包延迟
      │   │
      │   ├─ send_fn: upload_lwip_transport_send()
      │   │   └─ netconn_write(..., NETCONN_COPY)  // 阻塞写入整段数据
      │   │       └─ 等待 pcb->unsent == NULL  (确保整包已发出)
      │   │
      │   ├─ recv_fn: upload_lwip_transport_recv()
      │   │   └─ netconn_recv() → pbuf 遍历 → upload_http_save_response()
      │   │
      │   └─ close_fn: upload_lwip_transport_close()
      │       └─ netconn_close() → netconn_delete()
      │
      ├─ 结果处理:
      │   ├─ UPLOAD_HTTP_OK     → log_upload_success_function()  (删除 pending)
      │   └─ 其他错误码         → log_upload_fail_function()  (保留 pending)
      │
      └─ vTaskSuspend(NULL)  // 不自删除, 由 eth_task 调用 upload_lwip_delete() 回收
```

### 12.4 upload_http 内核: 三步分片上传协议

```
upload_http_file_function(request, transport, http_status_code)
  │
  ├─ 1) 根据 IP/端口构建 host 字符串
  │
  ├─ 2) 构建 HTTP 请求通用信息 (设备ID拼入 URI)
  │     └─ 路径: /fnwlw/oss/deviceLog?deviceId={HEX_ID}&folder=xx&fileType=json
  │
  ├─ 3) 计算文件大小 & 总分片数
  │     └─ log_open_pending_text_stream() 打开流
  │     └─ log_get_pending_text_size(&total_size)
  │     └─ total_chunks = (total_size + CHUNK_SIZE-1) / CHUNK_SIZE
  │     └─ log_close_pending_text_stream()
  │
  ├─ ★ 4) START 接口: 开始上传 ★
  │     └─ 构造 multipart/form-data POST 请求
  │         └─ 请求体: --boundary\r\nContent-Disposition...\r\n\r\n{json}\r\n--boundary\r\n
  │     └─ transport→connect(host, port)
  │     └─ transport→send(req, len)
  │     └─ 循环 transport→recv() 接收应答
  │         ├─ Transfer-Encoding: chunked → 等待结束块 0\r\n\r\n
  │         └─ Content-Length → 等待 body 完整
  │     └─ 解析 HTTP 状态码 + JSON "code" 字段
  │     └─ 提取 "data": { "uploadId": "xxx" }
  │
  ├─ ★ 5) CHUNK 接口: 循环发送每个分片 ★
  │     └─ log_open_pending_text_stream()
  │     │
  │     └─ for (chunk_seq = 1; chunk_seq <= total_chunks; chunk_seq++):
  │         │
  │         ├─ log_read_pending_text_stream(s_chunk, CHUNK_SIZE, &read_len)
  │         │
  │         ├─ 构造 multipart/form-data POST 请求:
  │         │   ├─ Content-Disposition:{uploadId, chunkSeq, chunkTotal, ...}
  │         │   └─ chunkData: 分片文件内容
  │         │
  │         ├─ transport→send(body, total_len)   // 发送整个 HTTP 请求体
  │         ├─ 循环 transport→recv() 接收应答
  │         └─ 解析应答 "code" 字段
  │         │   └─ code != 200 → 重试上传当前分片 (最多3次)
  │         │       └─ 每次重试重新 connect → send → recv
  │         │
  │         └─ (可选) 计算累计 MD5  (用于 FINISH 校验)
  │
  │     └─ log_close_pending_text_stream()
  │
  ├─ ★ 6) FINISH 接口: 完成上传 ★
  │     └─ 构造 multipart/form-data POST 请求
  │         └─ 请求体: {uploadId, md5(可选)}
  │     └─ transport→connect → send → recv
  │     └─ 解析 "code" 字段
  │     └─ transport→close()
  │
  └─ 返回: UPLOAD_HTTP_OK / 各种错误码
```

### 13.2 RS485 透传模式流程

```
dtu_task_function()
  │
  ├─ 初始化队列和信号量 (首次进入)
  │   ├─ dtu_config_sem = xSemaphoreCreateBinary()    // 配置更新信号量
  │   ├─ dtu_tx_q = xQueueCreate(1, sizeof(void*))     // 发送队列
  │   └─ dtu_rx_q = xQueueCreate(1, sizeof(void*))     // 接收队列
  │
  └─ for(;;) 主循环:
      │
      ├─ ★ 配置热更新检测 ★
      │   └─ if xSemaphoreTake(dtu_config_sem, 0) == TRUE:
      │       └─ RS485_ReConfig(&g_rs485_config)
      │           └─ 重新配置 UART7 波特率/数据位/校验位/停止位
      │
      ├─ ★ 发送路径: 平台指令 → RS485 总线 ★
      │   └─ if xQueueReceive(dtu_tx_q, &msg_ptr, 0):
      │       ├─ len = msg_ptr[0]                    // 第一字节=数据长度
      │       └─ rs485_send_str(msg_ptr+1, len)      // 通过 UART7 DMA 发送
      │           └─ RS485_GPIO_Init() 控制方向引脚 (发送模式)
      │
      └─ (MPPT 太阳能模式分支, 见 13.3)

外部模块调用 dtu_transmit_enqueue():
  │
  ├─ 来源: com_deal_configure_rs485_uostream()  // 平台下发 RS485 透传数据
  │   └─ com_rec_data_t.buff → dtu_transmit_enqueue(data, len)
  │
  └─ dtu_transmit_enqueue(data, len):
      ├─ dtu_tx_buf[0] = len;                     // 首字节存长度
      ├─ memcpy(dtu_tx_buf+1, data, len);         // 后续字节存数据
      └─ xQueueSend(dtu_tx_q, &dtu_tx_buf, 0);    // 非阻塞入队

RS485 接收 (ISR → 任务):
  │
  ├─ UART7 DMA 接收完成中断
  │   └─ rs485_get_receive_data(data, len)
  │
  └─ dtu_get_receive_data(data, len):
      ├─ dtu_rx_buf[0] = len;
      ├─ memcpy(dtu_rx_buf+1, data, len);
      └─ xQueueSendFromISR(dtu_rx_q, &dtu_rx_buf, &woken)
          └─ portYIELD_FROM_ISR(woken)
```

### 13.3 MPPT 太阳能模式流程

```
dtu_task_function() (MPPT 分支)
  │
  └─ for(;;) 主循环:
      │
      └─ ★ 6级状态机轮询 MPPT 控制器 (每60s一轮) ★
          │
          │  now = xTaskGetTickCount()
          │
          │  if step >= 2 && (now - timeout) > 2s:
          │      step = 0;  // ★ 超时保护: 等待响应超2s回退
          │
          │  if step == 0 && (now - last_tick) >= 60s:
          │      step = 1;  // ★ 60s 间隔触发新轮
          │
          ├─ step=1: 发送读电能参数命令
          │   └─ dtu_mppt_send_electricity()
          │       └─ DTU_MPPT_ReadRegisters(0x01, 0x04, 0x3045, 19)
          │           ├─ 构造 Modbus RTU 帧: [ID][0x04][AddrH][AddrL][CntH][CntL][CRCL][CRCH]
          │           └─ rs485_send_str(frame, 8)
          │   └─ step = 2; timeout = now;
          │
          ├─ step=2: 等待并解析电能参数响应
          │   └─ if xQueueReceive(dtu_mppt_rx_q, &rx_ptr, 0):
          │       └─ dtu_mppt_parse_electricity(rx_ptr+1, rx_ptr[0])
          │           └─ 解析 19 个 16-bit 寄存器 → g_mppt.realtime
          │               ├─ [0] battery_soc      (蓄电池 SOC)
          │               ├─ [1] battery_voltage  (蓄电池电压)
          │               ├─ [2] battery_current  (蓄电池电流)
          │               ├─ [3-4] battery_power  (蓄电池功率)
          │               ├─ [5-6] load_voltage/current/power
          │               ├─ [9-10] solar_voltage/current
          │               └─ [11+] 充电/放电能量统计
          │       └─ step = 3; (继续发下一个查询)
          │
          ├─ step=3: 发送读开关状态 → 解析白天/夜晚
          │   └─ dtu_mppt_send_02() → 0x02 功能码
          │   └─ dtu_mppt_parse_02() → g_mppt.ai_state
          │   └─ step = 4;
          │
          ├─ step=4: 发送读蓄电池状态 → 解析温度/保护计数
          │   └─ dtu_mppt_send_battery() → 0x04, 0x3032, 8 寄存器
          │   └─ dtu_mppt_parse_battery()
          │       └─ 提取: 电池等级/状态/充放电状态/温度/保护次数
          │   └─ step = 5;
          │
          └─ step=5: 发送读运行天数 → 解析 → 本轮结束
              └─ dtu_mppt_send_run_day() → 0x04, 0x309D, 1 寄存器
              └─ dtu_mppt_parse_run_day()
              └─ last_tick = now;  // 记录本轮完成时间
              └─ step = 0;  // 回到空闲, 等待下一次60s触发
```

## 十四、数据上报 -- 完整流程

### 14.1 上报体系概览

本项目数据上报采用 **消息队列 + 优先级发送** 架构，上报内容分为以下几类：

| 上报类型 | 触发方式 | 优先级 | 命令字 | 数据来源 |
|---------|---------|--------|--------|---------|
| **心跳包** | 定时器 (默认90s) | MSG_PRIO_NORMAL | COM_HEART_UPDATA (0xFF) | com_heart_pack_function |
| **周期上报** | 定时器 (默认180s) | MSG_PRIO_NORMAL | CR_QUERY_INFO (0xE2) | com_report_normally_function |
| **立即上报** | 事件触发 (alarm) | MSG_PRIO_HIGH | CR_QUERY_INFO (0xE2) | com_report_normally_function |
| **查询配置** | 平台请求 | MSG_PRIO_HIGH | CR_QUERY_CONFIG (0xE1) | com_query_configuration_function |
| **版本信息** | 平台请求 | MSG_PRIO_HIGH | CR_QUERY_SOFTWARE_VERSION | com_version_information |
| **IPC 信息** | 平台请求 | MSG_PRIO_HIGH | CR_QUERY_IPC_INFO | com_ipc_device_information |
| **SNMP 参数** | 平台请求 | MSG_PRIO_HIGH | CR_QUERY_SNMP_INFO | com_device_snmp_information |
| **Ping 信息** | 平台请求 | MSG_PRIO_HIGH | CR_QUERY_PING_INFO | com_device_ping_information |
| **ACK 回复** | 配置完成 | MSG_PRIO_HIGH | 原命令 | com_ack_function |

### 14.2 周期上报数据组包流程 (com_report_normally_function)

这是最主要的主动上报函数，将设备全部采集数据打包为一条文本帧。

```
com_report_normally_function(data, len, cmd)
  │
  ├─ ★ 帧头: "##" (2字节) + 长度占位(4字节)
  │
  ├─ ★ 请求编码 QN ★
  │   ├─ 平台请求时: QN={qn1}{qn2}  (带请求标识, 用于应答匹配)
  │   └─ 主动上报时: QN=0
  │
  ├─ ★ 设备标识 ★
  │   ├─ TID={device->id}           // 设备ID (十六进制)
  │   ├─ VER={COM_DATA_VERSION}     // 数据版本号
  │   └─ DEVTYPE={DEVICE_TYPE}      // 设备类型 (0x0400=YX市电)
  │
  ├─ ★ 指令参数 CP=&& ... && ★
  │   │
```

### 14.5 消息池生命周期

```
send_msg_t 生命周期:
  │
  ├─ ★ 分配: send_msg_alloc(prio, cmd, retries)
  │   └─ 遍历 g_msg_pool[8], 找到 in_use=0 的槽位
  │       ├─ memset(buf, 0, 2048)
  │       ├─ 设置 prio, cmd, retries
  │       ├─ dst = SEND_DST_AUTO (自动探测通道)
  │       └─ in_use = 1; return &pool[i];
  │   └─ 池满 → return NULL (调用者丢弃本次数据)
  │
  ├─ ★ 组包: 调用方直接写入 msg->buf, 设置 msg->size
  │
  ├─ ★ 入队: send_msg_enqueue(msg)
  │   └─ 按 prio 选择 g_high_q 或 g_normal_q
  │   └─ xQueueSend(q, &msg, 0)
  │   └─ 队列满 → 返回 -1 (消息丢失)
  │
  ├─ ★ 发送: send_task 出队 → send_data_task_function()
  │
  ├─ ★ 成功释放: send_msg_free(msg)
  │   ├─ msg->in_use = 0
  │   └─ g_sending = NULL
  │
  └─ ★ 失败重试: send_handle_fail(msg)
      ├─ retries_left > 0 → retries_left-- → send_msg_enqueue(msg)  (重新入队)
      └─ retries_left == 0 → send_msg_free(msg)  (丢弃)
```

| **OTA 升级** | 远端 HTTP 服务器 | update_lwip_task / update_gsm_task → HTTP GET → SPI Flash → Bootloader | MCU Flash |
| **日志上传** | log_device_write | LittleFS /log/current → 轮转 → /log/pending → upload_http_file → 分片上传 | HTTP 服务器 |
| **MPPT 数据** | RS485 Modbus | dtu_task → g_mppt → dtu_mppt_report_function → send_msg | 远端平台 |
| **告警上报** | alarm_task 检测 | 电气/网络/传感器异常 → app_report_information_immediately → send_msg(高优先级) | 远端平台 |
